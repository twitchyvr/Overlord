Core Principle

Don't Change the Agent — Change the Framework

The single architectural decision that makes everything else possible.

Rooms define behavior, not agents. When an agent enters a room, the room's rules, tools, and output templates merge into their context. Change the testing room rules once — every agent that enters inherits them. Add a new agent? It just works in existing rooms.

v1 Agent-Centric

Each agent carries 200+ line system prompts with behavioral rules. 15 agents = 15 places to maintain rules. Change one, miss the others.

v2 Framework-Centric

Agents are 10-line identity cards (name, capabilities, room access). Rooms own the rules. One change propagates to every agent that enters.

v2+ Scriptable

Rooms, floors, tables are plugins — scriptable modules (Lua or JS). Even built-in features are implemented as plugins, not monolith. Users can create custom room types for use cases we haven't imagined.

Phase Zero: The Strategist

Before discovery. Before anything. A consultative setup that builds your building for you.

New User (Empty Building)

You open Overlord for the first time. No projects, no agents, nothing.

The Strategist agent activates:

  • 1. "What are you trying to build/do?"
  • 2. "What does success look like?"
  • 3. "What inputs do you have? What outputs do you need?"
  • 4. "What constraints exist? (budget, timeline, tech)"
  • 5. Based on answers: suggests a building layout
  • 6. Quick Start: Accept suggested template
  • 7. Advanced: Drag-and-drop rooms into floors, create custom agents

Existing User (Full Building)

You open Overlord and have 5 projects, 20 agents, 50 completed phases.

The Lobby Dashboard shows:

  • + Project cards with pass/fail rates, open items
  • + Active agents and their current rooms
  • + Phase progress per project
  • + Pending approvals / gate reviews
  • + Quick action: "Start new project" → Strategist
  • + Quick action: "Resume project X" → active room
  • + RAID log search across all projects
Modes based on intent: "Brainstorming mode" = only Discovery floor. "Coding mode" = Execution + Testing floors. "Full project" = all floors. "Risk analysis" = feed docs, get Governance floor output. The Strategist figures out which floors to activate based on what you're trying to accomplish.

Phases & Gates

Each phase has an entry protocol, a working period, and an exit protocol. You cannot skip a gate.

Phase 0 Phase 1 Phase 2 Phase 3 Phase 4 Phase 5 STRATEGY G DISCOVERY G ARCHITECTURE G EXECUTION G REVIEW G DEPLOY | | | | | Exit Doc Exit Doc Exit Doc Exit Doc Exit Doc RAID Log RAID Log RAID Log RAID Log RAID Log Sign-off Sign-off Sign-off Sign-off Sign-off G = Gate (go/no-go checkpoint) Each gate requires: structured exit document + RAID log entry + sign-off Next phase receives previous phase's output as input template

Gate Protocol

At every gate, the following must be completed before advancing:

// Gate checklist (cannot be bypassed) { "phase": "discovery", "exitDocument": { "completedBy": "pm-agent", "decisionsLog": [...], // What was decided + rationale + evidence "risksIdentified": [...], // Each with analysis, not just yes/no "openItems": [...], // What's unresolved + who owns it "outputArtifacts": [...] // Links to docs, schemas, etc. }, "raidLogEntries": [...], // New RAID entries from this phase "signoff": { "reviewer": "architect-agent", "verdict": "GO", "conditions": [], "timestamp": "..." }, "nextPhaseInput": { ... } // Structured input for the next phase }

Exit Documents & RAID Log

"You can't just say 'done.' You have to show you actually thought about it." — the M&A risk questionnaire principle.

✘ v1: "It says it's done"

  • Agent says "Done!" with no structured proof
  • Tests pass because there are no tests for missing features
  • UI looks complete but buttons aren't wired up
  • No audit trail of decisions made during execution
  • No way to reference what was decided in earlier phases
  • Scope changes lose all context

✓ v2: Prescriptive Exit Protocol

  • Agent must complete structured exit document to leave room
  • Exit doc requires: decisions, rationale, evidence, citations
  • RAID log entry: Risks, Assumptions, Issues, Decisions
  • Every phase's exit doc becomes next phase's input
  • Searchable RAID log database — agents reference past decisions
  • Scope changes trigger re-entry to relevant rooms with full history

RAID Log Schema

// Searchable database: every agent references this before starting work { "id": "raid_001", "type": "decision", // risk | assumption | issue | decision "phase": "architecture", "roomId": "arch-room-1", "summary": "Use SQLite for MVP, migrate to Postgres later", "rationale": "Simplicity for phase 1, single-file deploy...", "decidedBy": "architect-agent", "approvedBy": "user", "affectedAreas": ["storage", "deployment"], "timestamp": "2026-03-10T...", "status": "active" // active | superseded | closed }

Mid-Stream Scope Changes

"What happens if we're in test phase but want to add a new feature?" — the answer is structured re-entry.

Scenario: Project is in TESTING phase. User wants to add a new feature. User: "I want to add search filtering before we deploy" | Orchestrator checks: 1. Current phase = Testing 2. Requested work = New feature (requires Discovery + Architecture + Execution) 3. Scope change detected | Scope Change Protocol: a. Query RAID log for all decisions made so far b. Package relevant decisions as context brief c. Re-enter Discovery Room with context brief attached d. Discovery room knows: "We are in test phase, adding scope" e. Discovery output = mini-requirements (scoped to new feature only) | f. Fast-track through Architecture (existing architecture + delta) g. Fast-track through Execution (scoped to new files only) h. Return to Testing (re-run all tests including new feature) | Result: New feature integrated without losing any previous context. All downstream dependencies updated via RAID log cascading.
Key principle: When scope changes, agents don't just "go build." They re-enter the relevant rooms with the full RAID log context brief. The brief is a condensed summary: "Here's what's been decided, what's been built, what's been tested. Now we're adding X." This prevents the AI from ignoring previous work.

Plugin & Scripting Architecture

Even built-in features are plugins. The core is just a bus + loader. Everything else plugs in.

Integration Floor = Plugin Boundary

All external communication goes through one floor. This keeps the building secure — you're not sending data out from 20 different points.

  • + Plugin Bay: Lua/JS scripts register here
  • + Data Exchange: webhooks, APIs, MCP
  • + Provider Hub: AI model adapters
  • + Strict typed I/O at the boundary
  • + Sandboxed execution (plugins can't access raw system)

What's Pluggable

  • Floor Define a new floor type with rooms, rules, access
  • Room Define a new room type with tools, output template
  • Tool Register a new tool with I/O contract
  • Agent Define a new agent type with capabilities
  • Provider Add a new AI provider adapter
  • Storage Swap storage backend (SQLite, Postgres, Redis)
  • Phase Define custom project phases

"When I want to add built-in stuff, I add a plugin just like an end user would. Then I'm building a core, not a monolith."

Standardization

I/O Contracts: "Every Module is an API"

Standardized input/output for every module, room, tool, and agent. If it plugs in, it has a contract.

Universal I/O Contract

// Every module, room, tool, agent, and plugin follows this pattern { "input": { "schema": { /* JSON Schema */ }, "required": ["field1", "field2"], "defaults": { "field3": "default_value" } }, "output": { "schema": { /* JSON Schema */ }, "required": ["result", "metadata"], "validate": /* Custom validation fn */ }, "error": { "code": "string", "message": "string", "retryable": boolean, "context": { /* debug info */ } } }

Content can change. Hierarchy stays consistent. Like a socket that always fits — plug anything into anything.

Agent System

Agents: Simple Identity, Room-Driven Behavior

v1 200-line prompts per agent

{ name: "testing-engineer", instructions: ` You are a testing engineer. NEVER modify source code... ... 200 lines of behavioral rules ... `, tools: [...], blockedTools: [...], }

v2 10-line identity cards

{ name: "testing-engineer", role: "QA Lead", capabilities: ["testing", "qa"], roomAccess: ["testing-lab", "review"], } // Room injects: tools, rules, output template // Agent = who. Room = what they can do.
One agent, one todo. Agents handle one discrete todo at a time. Todos are the atomic unit of work. "Task" means a milestone-level item that may have many todos. This terminology choice matters because AI models respond predictably to "todo" — they auto-create checklists and mark them off.

Provider-Agnostic AI Layer

Not locked to MiniMax or Claude. Swap providers per room, per task, per budget. Open source models welcome.

v1 MiniMax-Specific Hacks

Emoji-to-ASCII mapping, Unicode repair, synthetic tool IDs from text fallback parser, guardrail module — scattered across 6+ files because MiniMax behaves differently.

v2 Provider Adapter Pattern

One adapter file per provider. Internal format = Anthropic-native. Adapters translate at the boundary. Swap provider = swap one file.

ProviderUse CaseIntegration
Anthropic (Claude)Complex reasoning, architecture, planningNative format — zero translation
MiniMaxFast coding, high throughput, cost-effectiveAdapter: emoji fix, Unicode repair
OpenAIBroad capabilities, function callingAdapter: OpenAI ↔ Anthropic format
Local (Ollama, LM Studio)Offline, privacy, experimentationAdapter: OpenAI-compatible API
Open Source (Llama, Mistral)Custom fine-tuning, cost controlAdapter: model-specific quirks
Different models for different rooms. Discovery room might use Claude (complex reasoning). Code Lab might use a fast local model. Testing Lab might use a cheap model for repetitive test runs. Provider selection is per-room configurable.

Phase 0 — Stabilize

Merge Fixes & Tag v1 Baseline

Phase 1 — Foundation

Core Bus + Storage + Config

Thin event bus. Database schema (rooms, agents, messages, phase gates, RAID log, exit documents). Config service with validation. Swappable storage backend (start with SQLite, design for Postgres/Redis). No AI yet.

Room base class. Testing Lab room (simplest rules: no write tools). Exit document engine. Verify room constraints actually prevent tool access.

Phase 3 — Agents + AI

Provider-agnostic AI layer. Agent registry with room access. Room-scoped tool executor. First end-to-end: user → agent enters room → uses tools → exit doc → gate.

Discovery, Architecture, Code Lab, Review, Deploy rooms. Phase gates with sign-off. RAID log database. Scope change protocol. Full project lifecycle test.

Strategist agent for new users. Building Architect that configures floors/rooms based on consultative questions. Quick Start templates. Advanced drag-and-drop mode.

Phase 6 — UI

Floor/room/table visual layout (tycoon-style). Agent "chairs" at tables. Phase progress. RAID log viewer. Dashboard with project KPIs. Port component system from v1.

Lua/JS scripting layer. Plugin sandbox. Integration Floor (external APIs, webhooks). Slash commands, @-mentions, #room-references. Cross-room tagging. Docker-based dev stack.

Lessons Learned

What v1 Taught Us

▶

Bug API Error 2013: Orphaned tool_results

Cause: Tool denial paths injected bare assistant messages, breaking Anthropic's alternation. v2 fix: Append-only message history validated at insert. Impossible to create orphans by construction.

▶

Bug Orchestrator prompted for read-only tools

Cause: 25+ tools missing from tier registry, defaulting to "unknown." v2 fix: No tier registry. Room-scoped access = "is this tool in this room?" Binary, no confidence scores.

▶

Pattern "It says done but nothing works"

Cause: No structured proof of work. v2 fix: Exit documents require evidence (files modified, tests passed, citations). Risk questionnaire pattern — agents must show independent analysis, fill in the blanks, not just say "approved."

▶

Pattern Testing agent changes tests to pass

Cause: Agent prompt said "don't change tests" but the tool was available. v2 fix: Testing Lab room doesn't have write_file in its tool list. Structurally impossible, not just instructionally prohibited.

▶

Pattern Circular dependencies everywhere

Cause: Hub-spoke with no layering. v2 fix: Strict layer ordering: Transport → Rooms → Agents → Tools → AI → Storage. Each layer only depends on layers below.

▶

Pattern MiniMax-specific hacks in 6+ files

Cause: Provider-specific behavior scattered throughout. v2 fix: One adapter per provider. All quirks contained in the adapter. Rest of system speaks Anthropic-native only.

▶

Insight Scope changes lose all context

Cause: No structured record of decisions made. No RAID log. v2 fix: Every phase produces an exit document that goes into the RAID log. Scope changes trigger re-entry to relevant rooms with a context brief summarizing all previous decisions.

▶

Insight No phase gates = premature execution

Cause: AI just starts building immediately. No discovery phase. v2 fix: Phase Zero (Strategist) asks consultative questions first. Phase gates require structured sign-off. Can't enter Execution without Architecture sign-off.
